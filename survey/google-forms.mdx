---

Google Forms is not capture-only. With a one-time Apps Script setup, every submission is checked server-side and bad responses are deleted before they settle. The mechanism: respondents go through the HumanCheck first, VerifyYou returns them to a thin redirect page that reads the `?vyt=TOKEN` query parameter and forwards them to your Google Form with the token pre-filled into a hidden field. An installable Apps Script trigger then validates every submission and removes anything that does not pass.

There is a brief window between submission and deletion, but in practice bad responses rarely survive long enough to appear in a linked spreadsheet.

<Note>
The installable Apps Script trigger in Step 4 is the real enforcement layer. The response-validation regex in Step 3 is a browser-side UI convenience, not enforcement — it only blocks the rendered form's submit button. A direct POST or a crafted prefill submission bypasses it entirely. Do not skip Step 4.
</Note>

## Why a redirect page is needed

VerifyYou returns the respondent to `ORIGIN + RETURN_PATH` and appends `?vyt=TOKEN&vyc=1` to that URL. Google Forms uses a different query-parameter scheme for prefill: `?entry.XXXXXXX=VALUE`. These two formats cannot be the same URL without a translation step.

The redirect page is that translation step. It reads `?vyt` from the return URL and forwards the respondent to the Google Form with `?entry.XXXXXXX=VALUE` already populated. The respondent never sees this happen — it is a near-instant client-side forward.

A snippet for that redirect page:

```html
<!DOCTYPE html>
<html>
  <head>
    <script>
      const params = new URLSearchParams(window.location.search);
      const vyt = params.get("vyt") || "";
      const pid = params.get("PROLIFIC_PID") || ""; // or whichever pass_param you use
      const base = "https://docs.google.com/forms/d/e/FORM_ID/viewform";
      window.location.replace(
        `${base}?entry.XXXXXXX=${encodeURIComponent(vyt)}&entry.YYYYYYY=${encodeURIComponent(pid)}`
      );
    </script>
  </head>
</html>
```

Host this page anywhere reachable — a simple Vercel deployment, a GitHub Pages file, or a path on your own domain. Set its URL as the `return_path` in your VerifyYou Flow. Replace `FORM_ID`, `entry.XXXXXXX`, and `entry.YYYYYYY` with your actual values (see Step 2 for how to find the entry parameter names).

## Setup

<Steps>
  <Step title="Add short-answer fields for the token and participant ID">
    Add a short-answer question titled **Verification token** and mark it required. Add a second short-answer field for your participant identifier (Prolific PID, RID, etc.) if you are passing one through. Respondents never type into either field — the redirect page fills them automatically.

    To find the `entry.*` parameter names for your prefill URL, open your form, click the three-dot menu, choose **Get pre-filled link**, type a placeholder into each field, and copy the result. It looks like:

    ```
    https://docs.google.com/forms/d/e/FORM_ID/viewform?usp=pp_url&entry.XXXXXXX=PLACEHOLDER&entry.YYYYYYY=PLACEHOLDER
    ```

    The `entry.XXXXXXX` and `entry.YYYYYYY` values are unique to your form. Use them in the redirect snippet above and in Step 3.
  </Step>

  <Step title="Deploy the redirect page">
    Take the redirect snippet from the section above and host it at a stable URL. Set that URL as the `return_path` in your VerifyYou Flow configuration.

    Your `POST /v3/initialize` call will look roughly like:

    ```json
    {
      "verification_id": "your-flow-id",
      "origin": "https://your-redirect-host.com",
      "return_path": "/redirect",
      "pass_params": { "PROLIFIC_PID": "PARTICIPANT_ID_VALUE" }
    }
    ```

    `pass_params` carries the participant ID back on the return URL alongside the token — your redirect page reads it from `?PROLIFIC_PID=` and passes it into the form's prefill slot. Replace the `pass_params` key and value with whatever your recruitment platform provides. See [/recruitment/prolific](/recruitment/prolific) for Prolific-specific syntax and the signed entry-URL pattern.
  </Step>

  <Step title="Add response validation on the token field">
    Select the **Verification token** field, open its settings (three-dot menu on the field), and choose **Response validation**. Set the rule to **Regular expression** matching:

    ```
    ^vyt_[A-Za-z0-9_-]{20,}$
    ```

    This rejects a completely blank or obviously malformed value at the browser, before the form submits. It catches bad redirect wiring early and keeps junk out of the queue for the trigger to clean up.

    <Warning>
    This regex does not call VerifyYou, and it is a UI-layer convenience, never enforcement. It only blocks the rendered form's submit button in a browser. A direct POST to Google's form-response endpoint, or a crafted prefill submission, bypasses it entirely — the request never touches this validation rule. The Apps Script trigger in Step 4 is where real enforcement happens — do not skip it.
    </Warning>
  </Step>

  <Step title="Install the Apps Script trigger">
    Open your form, click the three-dot menu, and choose **Script editor**. This opens the bound Apps Script project for your form.

    Paste the snippet below, replacing the placeholder values:

    ```javascript
    const VY_PEEK_URL = "https://app.verifyyou.com/v3/survey/peek";
    const VY_KEY = "pk_val_YOUR_FLOW_KEY"; // your per-Flow publishable validation key
    const TOKEN_FIELD_TITLE = "Verification token"; // must match your field title exactly

    function onFormSubmit(e) {
      const response = e.response;
      const answers = response.getItemResponses();

      // Find the token field by its title
      let token = "";
      for (const answer of answers) {
        if (answer.getItem().getTitle() === TOKEN_FIELD_TITLE) {
          token = answer.getResponse();
          break;
        }
      }

      if (!token) {
        response.getForm().deleteResponse(response.getId());
        return;
      }

      const result = UrlFetchApp.fetch(
        `${VY_PEEK_URL}?vyt=${encodeURIComponent(token)}`,
        {
          method: "get",
          headers: { Authorization: `Bearer ${VY_KEY}` },
          muteHttpExceptions: true,
        }
      );

      if (result.getResponseCode() !== 200) {
        response.getForm().deleteResponse(response.getId());
        return;
      }

      const body = JSON.parse(result.getContentText());
      if (!body.verified) {
        response.getForm().deleteResponse(response.getId());
        return;
      }

      // Token is valid. Optionally burn it now to prevent reuse.
      // If you burn here, remove the consume call from any downstream webhook.
      // UrlFetchApp.fetch(
      //   `https://app.verifyyou.com/v3/survey/consume?vyt=${encodeURIComponent(token)}`,
      //   { method: "post", headers: { Authorization: `Bearer ${VY_KEY}` }, muteHttpExceptions: true }
      // );
    }
    ```

    Save the file. Then go to **Triggers** (the alarm-clock icon on the left sidebar), click **Add Trigger**, and configure:

    - **Function:** `onFormSubmit`
    - **Event source:** From form
    - **Event type:** On form submit — pick the **installable** trigger, not the simple one

    The installable trigger runs with your Google account's permissions. That is what grants it access to `deleteResponse`. The simple `onSubmit` trigger does not have that permission and will silently fail to delete anything.

    Each submission triggers one `UrlFetchApp` call and finishes well inside Apps Script's limits — 20,000 URL Fetch calls per day on a personal Google account (100,000 on Google Workspace) and a 6-minute cap per trigger execution — far more headroom than any realistic study needs.

    <Info>
    `peek` is idempotent — it does not burn the token. If you also process submissions from a linked spreadsheet or a second Apps Script, you can call `peek` again safely. Call `consume` exactly once, when you are ready to mark the response as final and prevent any further use of that token.
    </Info>
  </Step>
</Steps>

## What each side ends up with

A respondent without a valid token cannot leave a lasting response. The trigger fires on every submission, calls `peek` against the VerifyYou validation endpoint, and calls `deleteResponse` on that one response if it fails — `deleteResponse` takes a single response ID per call, so each bad submission is removed individually as it comes in, not swept up later in a batch. From your spreadsheet's point of view, bad submissions either never appear or are gone within seconds of landing. You get a clean dataset — every row a real, unique, live human — without touching the responses manually.

If you think of your screener as the list of criteria a respondent must pass before their answer counts, the HumanCheck is one more criterion on that list. The trigger enforces it automatically. It does not replace your other screener questions; it sits alongside them.

This is the same enforcement tier as the Tally and Jotform webhook flows — soft gate, auto-reject on submission — not a true pre-content gate like Qualtrics personal links or Alchemer login actions.

## Trade-offs

  <Card title="What works well" icon="circle-check">
    No platform fee, no API upgrade, no paid plan required. Works with any Google Workspace or personal Google account. The installable trigger runs server-side with your credentials — it is not bypassable from the browser. Deletion is fast enough that bad responses rarely surface in a linked spreadsheet, and the setup is a one-time copy-paste.
  </Card>
  <Card title="Honest limits" icon="triangle-exclamation">
    There is a brief window between submission and deletion where the response technically exists. It will not appear in exports if deletion is prompt, but this is not a true gate — no question is blocked before it loads. If you need zero exposure, use a Qualtrics personal link or an Alchemer login action instead. See [/choosing-your-flow](/choosing-your-flow).
  </Card>
  <Card title="Apps Script quota" icon="clock">
    Google Apps Script caps `UrlFetchApp` calls at 20,000/day on a personal Google account, 100,000/day on Google Workspace, with a 6-minute limit per trigger execution. That is far above any realistic study volume, but very high-volume studies should check their usage under Quotas in the Apps Script dashboard. For very large studies, a true-gate platform is more reliable and removes the quota concern entirely.
  </Card>
  <Card title="No failure redirect yet" icon="arrow-right-arrow-left">
    VerifyYou does not yet have a built-in failure redirect. Respondents who do not pass the liveness check return to your redirect page with `vyc=0` and a token that will not validate. The Apps Script trigger handles this correctly — it deletes the response — but the respondent experience after a failed check depends on how your recruitment platform handles a screen-out.
  </Card>

## `peek` vs `consume`

| Call | Burns token? | When to use |
|---|---|---|
| `GET /v3/survey/peek?vyt=TOKEN` | No | Checking validity at submission, or checking again from a second script |
| `POST /v3/survey/consume?vyt=TOKEN` | Yes, once | When you are ready to accept the response as final and prevent any reuse |

For most Google Forms setups, calling `peek` in the trigger and deleting bad responses is enough. If you want to prevent a token from passing a second submission — say, a respondent re-opens the confirmation page and hits Submit again — uncomment the `consume` call in the snippet once you are confident the response is good.

## Testing

Before you send your recruitment link to respondents, verify all four cases:

1. **Blank token.** Submit the form with the token field empty (bypass the redirect). The response should be deleted and not appear in your linked spreadsheet.
2. **Plausible but invented string.** Submit with something like `vyt_FAKEFAKEFAKEFAKEFAKEFAKE`. Same result: deleted.
3. **Real token in test mode.** Complete a real HumanCheck in test mode to get a genuine token. Submit the form with that token. The response should persist and `peek` should return `verified: true`.
4. **Token reuse without burning.** Submit a second time with the same token from case 3 (without calling `consume`). Because `peek` is idempotent and the token has not been burned, the response should persist again. If you called `consume` in case 3, this second submission should be deleted.

Run these tests against your live form before you field.
