Bad completes sitting in your Tally dataset are not a counting problem. They are a data-quality problem. Two respondents arrive; one is real, one is not. Your export does not know the difference. The HumanCheck is what makes that distinction before the data ever lands.

Tally gives you two shapes to work with. Which fits depends on how strictly you need to block unverified respondents before they see anything.

<CardGroup cols={2}>
  <Card title="Wrapper page — true gate" icon="shield-check">
    Verification runs before the form opens. An unverified respondent never sees a single question. The form itself stays invisible until the HumanCheck passes.
  </Card>
  <Card title="Hidden field + webhook delete — soft gate" icon="webhook">
    The form shell loads. A webhook fires on every submission, checks the token, and calls the Tally API to delete any response that fails. No bad complete survives in your dataset.
  </Card>
</CardGroup>

## Option A: wrapper page (true gate)

A small page you host calls `Tally.openPopup()` only after the HumanCheck passes. The form never opens for anyone who has not cleared verification.

<Steps>
  <Step title="Create your Tally form and note the form ID">
    Build your form in Tally. The form ID is the short code after `tally.so/r/` in the share URL. Enable the **Embed** option and copy the popup embed snippet — that gives you the `Tally.openPopup('FORM_ID')` call you will fire conditionally.
  </Step>
  <Step title="Initialize a VerifyYou session from your server">
    Call `POST /v3/initialize` with your secret key:

    ```json
    {
      "verification_id": "your-flow-id",
      "pass_params": { "RID": "PARTICIPANT_ID" },
      "origin": "https://your-wrapper-page.com",
      "return_path": "/verified"
    }
    ```

    The response returns a `url` pointing to `app.verifyyou.com/v/SESSION`. Send the respondent there.
  </Step>
  <Step title="Read the return and peek the token before opening the form">
    After the check, VerifyYou returns the respondent to `origin + return_path` with `?vyt=TOKEN&vyc=1` (verified) or `vyc=0` (not). On your return page, read the query string. `vyc` is a UI hint only — confirm the token server-side before trusting it.

    If `vyc` is not `1`, show a screen-out message and stop. If it is, confirm the token server-side before opening anything:

    ```http
    GET /v3/survey/peek?vyt=TOKEN
    Authorization: Bearer pk_val_FLOW_RAND
    ```

    `peek` is idempotent — it does not burn the token. A `{ "verified": true }` response means the token is real and still unused. Only then call `Tally.openPopup('FORM_ID')`.
  </Step>
  <Step title="Pass the token into the form as a hidden field">
    Open the popup with the token and participant ID in `hiddenFields`, so they travel through to the submission:

    ```js
    Tally.openPopup('FORM_ID', {
      hiddenFields: {
        vyt: token,
        rid: participantId
      }
    });
    ```

    Map `vyt` to a hidden field you have already created in your Tally form (see Tally's embed docs for the hidden fields spec).
  </Step>
  <Step title="Consume the token when a clean submission arrives">
    When Tally fires your `FORM_RESPONSE` webhook for a completion you want to count, burn the token exactly once:

    ```http
    POST /v3/survey/consume?vyt=TOKEN
    Authorization: Bearer pk_val_FLOW_RAND
    ```

    This locks the token so the same verified session cannot be replayed into a second submission.
  </Step>
</Steps>

<Tip>
  The `peek` at the wrapper page is the gate. The `consume` on submission is the second lock. Together they mean a respondent cannot save the return URL and re-submit later, even if they somehow reload the wrapper.
</Tip>

## Option B: hidden field + conditional logic + webhook delete (soft gate)

No wrapper page. Verification happens through a Tally hidden field, a logic rule that routes an empty token to a disqualification screen, and — critically — a webhook that calls the Tally delete API to remove any submission that fails the token check. The delete call is what separates this from capture-only.

<Warning>
  With this option, the first page of your form loads before verification is confirmed. An unverified respondent sees that shell before being screened out. If even that brief exposure is a problem for your study design, use Option A.
</Warning>

<Steps>
  <Step title="Add a hidden field named vyt">
    In your Tally form, add a **Hidden field** and name it `vyt`. In your VerifyYou dashboard, set the **Success redirect URL** to your Tally form share link. VerifyYou appends `?vyt=TOKEN` automatically on redirect, and Tally picks up query parameters that match hidden field names without any extra configuration.
  </Step>
  <Step title="Add a conditional logic disqualification screen">
    Add a screen early in your form — right after any intro page, before any real questions — that contains only a disqualification message.

    In the logic builder, add a rule: if `vyt` **is empty**, jump to the disqualification screen. This fires when a respondent clicks Next, catching anyone who arrived without passing through the VerifyYou link.

    <Note>
      Tally's conditional logic docs cover conditions on input blocks; a hidden field driving the jump is not explicitly documented, so test this rule in the builder before relying on it in production. If it does not work as expected, the webhook-delete backstop in Step 4 still catches any submission that lacks a valid token, so you are covered either way.
    </Note>
  </Step>
  <Step title="Enable Prevent duplicate submissions on the vyt field">
    In your Tally form settings (**Settings > Access**), turn on **Prevent duplicate submissions** and select `vyt` as the uniqueness key. Tally can key uniqueness on any field you choose, including a hidden one, so `vyt` works. This blocks a resubmission of the same token value at the platform level, before the webhook even fires.

    Two caveats worth knowing. First, this only blocks a *replay of the same value* — a respondent who fabricates a different `vyt` string sails past the duplicate check and gets caught downstream by the webhook peek instead. Second, Tally only checks submissions made after you flip the toggle on; anything submitted earlier is not retroactively deduplicated.
  </Step>
  <Step title="Set up the FORM_RESPONSE webhook — and make sure it deletes, not just flags">
    This is the step that turns a soft gate into a soft-gate auto-reject. In Tally integrations, add a webhook pointing to your validation endpoint. When Tally calls your endpoint on `FORM_RESPONSE`, your server must:

    1. Extract the `vyt` value from the submission's `fields` array — hidden fields come through in the `FORM_RESPONSE` payload with `type: "HIDDEN_FIELDS"`.
    2. Call `GET /v3/survey/peek?vyt=TOKEN` (with `Authorization: Bearer pk_val_FLOW_RAND`) to check the token.
    3. If `verified` is `false`, or the token is missing, call the Tally API to delete the submission immediately:

    ```http
    DELETE https://api.tally.so/forms/FORM_ID/submissions/SUBMISSION_ID
    Authorization: Bearer YOUR_TALLY_API_KEY
    ```

    A successful delete returns `204 No Content`.

    4. If `verified` is `true`, call `POST /v3/survey/consume?vyt=TOKEN` to burn it, then proceed with normal completion handling.

    Without the DELETE call, bad submissions sit in your Tally dataset. You may have flagged them, but they are still there. The delete is the upgrade that makes this an auto-reject instead of a capture-only record — and the difference matters when you export data or count completes.
  </Step>
</Steps>

## What each option gives you

| | Wrapper page (A) | Hidden field + webhook delete (B) |
|---|---|---|
| **When the gate fires** | Before the form opens | After submission, via webhook |
| **Respondent exposure** | None — form invisible until verified | Form shell loads; disqualification screen on page turn |
| **Setup you host** | A wrapper page + server endpoint | A webhook endpoint only |
| **Duplicate block** | Token consumed on webhook receipt | Tally duplicate-prevention on `vyt` + token consume |
| **Best for** | Studies where any form-shell exposure is unacceptable | Lower-friction setup when brief shell exposure is fine |

## Token security

Both options use the same two-call pattern for a reason. `peek` is idempotent — you can call it at the entry gate and again from the webhook without burning the token. `consume` is single-use — call it exactly once when you accept a completion. The **publishable validation key** (`pk_val_`) is safe to store in your webhook handler or Tally's credential store: it can check and burn tokens for one specific Flow, but it cannot create sessions, read PII, or reach across Flows.

Do not skip authentication on either call. An unauthenticated peek endpoint lets anyone probe whether a token string is valid. An unauthenticated consume lets a determined respondent burn tokens at will.

<AccordionGroup>
  <Accordion title="What happens if my webhook server is down when Tally fires?">
    Tally retries failed webhook deliveries. Until your server responds with a success status, the deletion has not happened and the submission sits in your dataset. For critical studies, log every webhook receipt and every Tally delete call so you can manually clean up any submissions that slipped through a transient outage.
  </Accordion>
  <Accordion title="Can the same respondent get a second token and submit again?">
    No. VerifyYou's uniqueness check means a respondent who has already passed cannot earn a second token for the same Flow — their face vector is already in the uniqueness pool. The `consume` call on the first valid submission also burns the token, so even if a second form session were somehow opened, the token would fail the peek on the next webhook.
  </Accordion>
  <Accordion title="The vyt logic rule is not appearing in the Tally logic builder">
    Confirm the hidden field named `vyt` exists in the form before opening the logic builder. Tally populates the field list dynamically from fields already in the form. If the field was added after you last opened the logic panel, close and reopen it.
  </Accordion>
  <Accordion title="Is there a screen-out redirect for respondents who fail the check?">
    A dedicated failure redirect URL is on the VerifyYou build list but does not exist yet. In the meantime, handle `vyc=0` on your return page (Option A) or rely on the conditional logic disqualification screen (Option B). Either way, the respondent sees a screen-out; they are not left on a blank page.
  </Accordion>
</AccordionGroup>
