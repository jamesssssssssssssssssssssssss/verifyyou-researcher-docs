SurveyMonkey is not capture-only. With **Advanced Branching** — available on Premier, Team Premier, and Enterprise plans, distinct from basic skip logic — you can read a custom variable on page 1 and disqualify a respondent on the first page turn, before your question blocks render. Pair that with a `response_completed` webhook and a Response DELETE, and bad tokens never become counted, paid completes. The page-1 branch is your soft gate. The webhook delete is the backstop.

<Note>
Advanced Branching and basic skip logic are not the same thing. Basic skip logic cannot read custom variables. Advanced Branching can, and it is required for the page-1 screen-out described here. Advanced Branching is available on Premier, Team Premier, and Enterprise plans.
</Note>

## How it works

The flow follows the same two-leg shape as every other platform: recruitment source on one side, your survey on the other, and the HumanCheck sitting in the middle. The two sides never see each other.

1. The respondent clicks your recruitment link — from a panel, Prolific, an email, wherever you source sample.
2. They pass through the HumanCheck first: liveness (a live person, not a photo, replay, or generated face), then uniqueness (never seen this face before in this Flow).
3. On a pass, VerifyYou returns them to your SurveyMonkey collector URL with `?vyt=TOKEN` appended. You set this as the **success redirect** in the VerifyYou dashboard.
4. SurveyMonkey picks up `vyt` as a custom variable on entry.
5. Advanced Branching on page 1 reads that variable. If it is empty or malformed, the respondent is routed to a DISQUALIFY outcome with a redirect — before any question block renders.
6. Respondents who clear the branch complete the survey. Your `response_completed` webhook fires, you peek the token server-side, and if it fails you DELETE the response via the SurveyMonkey API.

A respondent without a token hits the page-1 wall immediately. One who somehow carries a bad token through is caught and deleted before the response is kept.

## Step-by-step setup

<Steps>

<Step title="Create your VerifyYou Flow">
In the VerifyYou dashboard, create a Flow for this study. Copy the **Flow ID** and generate a **pk_val_ validation key** — you will need it in the webhook step. Set the **success redirect URL** to your SurveyMonkey collector URL.

There is no failure redirect in the dashboard yet. Respondents who fail liveness or uniqueness remain on the VerifyYou result screen. Build that expectation into your recruitment messaging.
</Step>

<Step title="Add the vyt custom variable">
In your survey's **Custom Variables** settings, add a variable named `vyt`. Custom variables only work with the **Web Link collector** — if your collector is email, or another type, this approach does not apply. You can have up to 100 custom variables per survey.

SurveyMonkey does not populate the variable itself: the value is passed on the survey URL as a query parameter (`?vyt=TOKEN`), which your return URL carries from the VerifyYou redirect. No visible field is required. It travels as a hidden collector variable.
</Step>

<Step title="Add Advanced Branching on page 1">
On page 1 of your survey, before any question block, add an Advanced Branching rule:

- **Condition:** custom variable `vyt` is empty (or does not match a basic format check if you want to be stricter).
- **Action:** disqualify and redirect to a screen-out page of your choice.

Anyone who navigates directly to the survey URL — or arrives from a source that skipped the HumanCheck — hits this branch immediately. No questions are served.

<Warning>
Basic skip logic cannot read custom variables. If you are not on Premier, Team Premier, or Enterprise, the page-1 branch is unavailable, and respondents can see and answer questions before the webhook removes their response. If you need a true pre-content gate without a plan upgrade, see [Qualtrics](/survey/qualtrics) or [Alchemer](/survey/alchemer).
</Warning>
</Step>

<Step title="Wire the response_completed webhook">
In SurveyMonkey's **Webhook** settings, add a `response_completed` webhook pointing to an endpoint you control — your own server, a Vercel function, a simple proxy.

When the event arrives:

1. Extract the response ID and the `vyt` custom variable value from the payload.
2. Call `GET /v3/survey/peek?vyt=TOKEN` with your `pk_val_` key in the `Authorization: Bearer` header.
3. If `verified` is `false` or the token is missing, call `DELETE /v3/surveys/{survey_id}/responses/{response_id}` on the SurveyMonkey API to remove the response.
4. If `verified` is `true`, call `POST /v3/survey/consume?vyt=TOKEN` to burn the token and record the final completion.

```http
GET /v3/survey/peek?vyt=TOKEN HTTP/1.1
Authorization: Bearer pk_val_FLOW_RAND
```

```json
{ "verified": true, "reason": "ok" }
```

```http
DELETE https://api.surveymonkey.com/v3/surveys/{survey_id}/responses/{response_id}
Authorization: Bearer YOUR_SM_ACCESS_TOKEN
```

<Note>
`peek` is idempotent and does not burn the token. `consume` burns it exactly once. Call `consume` on a verified response to mark the completion and prevent token reuse after the TTL window expires.
</Note>

<Note>
The DELETE endpoint requires your public app to hold the **Create/Modify Responses** OAuth scope. API access itself is plan-gated by SurveyMonkey, but this endpoint carries no additional gating beyond that scope.
</Note>
</Step>

<Step title="Pass your participant ID through">
If you are recruiting from a panel — Prolific, Lucid, or similar — pass the participant ID through using `pass_params` in your initialization call. VerifyYou appends it to the return URL alongside `?vyt=`, so SurveyMonkey receives both the token and the participant identifier in the same landing URL.

```http
POST /v3/initialize HTTP/1.1
Content-Type: application/json
Authorization: Bearer pk_YOUR_KEY

{
  "verification_id": "flow_abc123",
  "origin": "https://yoursite.com",
  "return_path": "/redirect-to-survey",
  "pass_params": { "PROLIFIC_PID": "{{%PROLIFIC_PID%}}" }
}
```

Your return page at `origin + return_path` receives `?vyt=TOKEN&PROLIFIC_PID=...` and forwards the respondent to your SurveyMonkey collector with both parameters. Map `PROLIFIC_PID` to a second custom variable in SurveyMonkey if you want it in your response data.
</Step>

</Steps>

## What each plan gets you

<CardGroup cols={2}>

<Card title="Soft gate (Premier, Team Premier, Enterprise)" icon="shield-halved">
Advanced Branching reads the `vyt` variable on page 1 and disqualifies before any question loads. Requires Premier, Team Premier, or Enterprise. This is the recommended setup.
</Card>

<Card title="Webhook backstop (all plans)" icon="rotate">
`response_completed` fires, you peek the token, and delete the response via the API if it fails. Available on every plan. Respondents may see questions before rejection, but the response is never kept.
</Card>

<Card title="What is stopped regardless" icon="circle-xmark">
Repeat takers are stopped at the uniqueness layer. The same face cannot earn a second token, so they cannot re-enter the Flow — regardless of which enforcement tier you use.
</Card>

<Card title="The honest catch" icon="triangle-exclamation">
The page-1 branch checks whether `vyt` is present, not whether it is genuinely valid. Full token validation happens in the webhook. Without Premier, Team Premier, or Enterprise, the webhook is your only enforcement line.
</Card>

</CardGroup>

## Questions teams ask

<AccordionGroup>

<Accordion title="Is SurveyMonkey capture-only?">
No — that is the common misread. Basic skip logic is capture-only because it cannot read custom variables. Advanced Branching (Premier, Team Premier, and Enterprise plans) can, and it disqualifies on page 1 before any question renders. That is not capture-only; that is a soft gate. The webhook DELETE is then a backstop, not the primary mechanism.
</Accordion>

<Accordion title="What if a respondent shares their token URL with someone else?">
The token has a short TTL (~30 minutes) and is minted per unique face. Within that window, a shared URL carrying a valid token would pass the page-1 presence check — the branch only confirms `vyt` is non-empty, not that it is genuine. Full validation happens in the webhook: `peek` confirms the token is real and `consume` burns it once, so the first completion locks the token and a second use of the same URL fails validation. The real backstop is calling `consume` promptly on every verified completion. Repeat *takers* are stopped at the uniqueness layer — the same face cannot earn a second token — but the sharing window is a real gap. Token TTL and `consume` are your defenses. If you need stronger entry-time validation, see [Qualtrics](/survey/qualtrics) via single-use personal links.
</Accordion>

<Accordion title="Can I use this without a Premier plan?">
Yes, but with a degraded experience. The webhook-and-delete backstop works on all plans. Respondents will see your questions before their response is removed. If that exposure is a problem — either for data integrity or for respondent experience — upgrade to Premier, Team Premier, or Enterprise, or use a platform that gates at entry without a plan requirement: [Qualtrics](/survey/qualtrics) via single-use personal links, or [Alchemer](/survey/alchemer) via one-time passwords.
</Accordion>

<Accordion title="Does the webhook fire on a disqualified (branched-out) respondent?">
No. SurveyMonkey's `response_completed` webhook fires when a respondent reaches the end of the survey. A respondent disqualified by the page-1 branch never completes; no webhook fires for them. The webhook is therefore acting on completes only — which is exactly what you want the backstop to cover.
</Accordion>

</AccordionGroup>
