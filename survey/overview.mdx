The survey leg is where your screener's human check does its work. You have verified that the respondent is a real, unique, live human. Now the question is: how strongly does your survey tool enforce that verdict?

Some tools let you block the door before a single question loads. Others delete a bad submission the instant it arrives. A few let your own code call us before anything is saved at all. The platform determines the tier; the tier determines the window.

Underneath all of it: uniqueness backstops everything. A respondent who already passed cannot earn a second token for the same study. The repeat-taker problem is stopped at the source regardless of when your platform check fires. See [Platform tiers](/platform-tiers) for the full mechanics.

## The three enforcement tiers

Think of these as *when* a bad respondent is caught, not *whether*.

<CardGroup cols={3}>
  <Card title="True gate" icon="shield-halved">
    The credential is the gate. Only a respondent who passed gets a valid entry path — a personal link or a one-time password — so no one without a real token ever sees a question.
  </Card>
  <Card title="Soft gate + webhook-delete" icon="arrows-rotate">
    The token is checked the instant the respondent submits. A bad submission is deleted via the platform API before it lands in your dataset, your CSV, or your client deliverable.
  </Card>
  <Card title="Direct call" icon="code">
    Your code calls us before saving anything. Nothing is persisted until the check passes. Functionally a true gate.
  </Card>
</CardGroup>

## Or start passive

Don't see your platform below, or not ready to wire up any of this yet? You don't need an integration to start. On any platform — including ones with no integration story at all — you can run the check as a mid-survey step and cross-reference the results afterward: export a verified/not-verified column and join it to your data on session or participant ID.

It is not a gate. Bad respondents still submit, and you filter them out after the fact rather than before. But it needs zero platform work, so it is the fastest way to run your first study while you decide which tier to build toward. See [The Screener](/the-screener) for the full passive flow.

## Platform comparison

| Platform | Tier | Family | Best flow |
|---|---|---|---|
| Qualtrics | True gate | A — single-use credential | Invitation-only distribution with single-use personal links |
| Alchemer | True gate | A — single-use credential | Login/Password Action loaded with one-time passwords |
| Typeform | Soft gate + webhook-delete | B — wrapper + webhook-delete | Wrapper page holds the link; webhook auto-deletes bad submissions |
| Jotform | Soft gate + webhook-delete | B — wrapper + webhook-delete | Wrapper page holds the link; webhook auto-deletes bad submissions |
| Tally | Soft gate + webhook-delete | B — wrapper + webhook-delete | Wrapper page holds the link; webhook must DELETE (not just flag) |
| SurveyMonkey | Soft gate | B — platform branching | Advanced Branching on a custom var disqualifies on page 1 before capture |
| Google Forms | Soft gate + webhook-delete | B — wrapper + webhook-delete | Wrapper page holds the link; Apps Script deleteResponse removes bad rows |
| Formspark | Direct call | D — direct call | JS calls peek before submit fires; bad attempts never reach the server |
| Your own form | Direct call | D — direct call | `GET /v3/survey/peek` server-side before saving; `POST /v3/survey/consume` on acceptance |

<Note>
MTurk sits on the recruitment side, not here. It enforces via a custom Qualification requirement — a true gate at the point of task acceptance, before the respondent ever clicks your link. See [Recruitment: MTurk](/recruitment/mturk).
</Note>

## How the token moves through the survey leg

After the respondent passes their liveness and uniqueness check, we return them to your redirect URL with a token on the query string:

```
https://yourdomain.com/wrapper?pid=abc123&vyt=tok_live_...&vyc=1
```

`pid` is carried back via `pass_params` — you set that when you initialize the session so your participant ID travels with the respondent on the return. `vyt` is the token. `vyc=1` is a UI hint only — confirm server-side via the token, never trust `vyc` alone. Then your tool enforces one of three ways:

**True gate tools (Qualtrics, Alchemer).** The token never needs to be read by the platform itself. The gate is the credential: a personal link, a one-time password. Only a respondent who passed gets one. The platform enforces by controlling who has a valid entry path at all.

**Soft gate tools (Typeform, Jotform, Tally, Google Forms).** The token rides in a hidden field or a prefilled question. For Typeform specifically, this requires a wrapper page as the redirect target, not a direct link straight into Typeform — hidden fields ride the `#` fragment on direct Typeform links, and the embed SDK only picks them up via its hidden-field option, so your server-side wrapper is what reads `vyt` from the query string and forwards it into Typeform correctly. On submission your webhook calls `GET /v3/survey/peek` and, if verified, calls `POST /v3/survey/consume` to burn it. If either check fails, delete the submission via the platform API.

**SurveyMonkey.** Advanced Branching on a custom variable disqualifies a bad respondent on page 1, before they reach your questions proper. This is platform-native enforcement rather than a webhook-delete — bad respondents are terminated by the survey logic itself before capture.

**Direct-call tools (Formspark, your own form).** Your code reads the token before saving anything and calls peek, then consume, inline.

### The two survey endpoints

For soft-gate and direct-call flows, use these two endpoints in sequence:

```http
GET /v3/survey/peek?vyt=TOKEN
Authorization: Bearer pk_val_FLOW_RAND
```

Returns `{ "verified": true, "reason": "ok" }` or `verified: false` with a reason. Idempotent. Does not burn the token. Safe to call at an entry gate and again from a webhook.

```http
POST /v3/survey/consume?vyt=TOKEN
Authorization: Bearer pk_val_FLOW_RAND
```

Returns `{ "consumed": true }`. Burns the token. Call this exactly once, when you have a response you intend to keep.

<Tip>
Call peek at entry (if you control the landing page) and again at submission. Call consume only at submission, after you have decided to accept the response. You get a true-gate check at entry and a webhook safety net at submit, with the token burned once on a genuine completion.
</Tip>

<Warning>
Always authenticate with your per-Flow publishable validation key (`pk_val_...`) as a Bearer token. Do not make unauthenticated calls. An open endpoint would let an attacker probe whether any token string is valid — and consume tokens they did not earn.
</Warning>

The publishable validation key is validate-only — it carries no PII and is safe to use server-side. It is not your secret `sk_` key.

## Choosing your platform

If you have flexibility and the study requires maximum enforcement, Qualtrics or Alchemer is the answer. The credential is the gate. There is no window where a bad respondent has seen any questions.

Already on Typeform, Jotform, Tally, or Google Forms? The wrapper-plus-webhook-delete pattern gets you to soft gate. Every bad submission is deleted before it counts. The honest difference from a true gate is a narrow window between when the form loads and when the respondent submits — during which a determined bad actor has seen your questions. For most survey purposes this window is acceptable.

On SurveyMonkey? Advanced Branching terminates a bad respondent on page 1 via the survey's own logic, before they see your core questions.

Building your own form or using Formspark? A direct call gives you true-gate behavior with full control over the implementation.

<AccordionGroup>
  <Accordion title="My platform is not in this list. Which family applies?">
    The families are broad. If your tool supports single-use access links or one-time login credentials, treat it as Family A. If it supports webhooks on submission and a deletion API, treat it as Family B. If you control the server-side code that saves responses, treat it as Family D. If none of those apply, reach out and we can advise on the right pattern for your setup.
  </Accordion>
  <Accordion title="Can I use the no-code share link?">
    The no-code share link (`app.verifyyou.com/verification/ID`) plus a success Redirect URL in the dashboard is the simplest path when you only need to confirm a respondent passed and redirect them onward. It does not give you token-level enforcement inside the destination: your survey tool receives `vyt` on the return URL but does not check it unless you wire the endpoints. Use the share link when the redirect itself is your gate — for example, landing on a Qualtrics personal link after passing. Note: a failure redirect does not exist yet; on a fail the respondent returns with `vyc=0` and your return page routes the screen-out.
  </Accordion>
  <Accordion title="What if the token expires before the respondent submits?">
    Tokens have a TTL of about 30 minutes from when we issue them. If a respondent takes longer than that and then submits, peek returns `verified: false` with an expiry reason. Treat this the same as any other failed check: delete the submission. A 30-minute window covers the large majority of completions. Design the flow so respondents enter the survey immediately after passing rather than queuing the link for later.
  </Accordion>
  <Accordion title="What does 'webhook must DELETE, not flag' mean for Tally?">
    On some platforms it is tempting to mark a bad submission with a flag or a tag and filter it out in analysis. That leaves the response in your dataset. Delete it entirely via the platform API so it cannot make its way into a CSV export, an automated report, or a downstream integration. The HumanCheck is not a filter column — it is a gate.
  </Accordion>
</AccordionGroup>
