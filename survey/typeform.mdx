title: "Typeform"
description: "Gate a Typeform survey so only verified, unique humans can reach it \u2014 the VerifyYou wrapper validates the token before forwarding anyone to your form, and in-survey logic plus a webhook catch anything that slips through."
---

Typeform's built-in logic can jump to an end screen. What it cannot do is block the form from loading in the first place. A welcome screen renders to everyone; the jump fires after. By the time an unverified respondent hits a dead end, they have already seen your questions — and a determined one can strip the URL parameter and bypass the jump entirely.

The setup below adds two layers on top of that: a server-side wrapper that validates the token and forwards only verified respondents to your survey, and webhook-plus-delete to catch anything that still lands without a valid token. Together they close the gap that Typeform's native logic leaves open.

<Info>
Typeform sits in the **soft gate** tier. The VerifyYou wrapper validates server-side before forwarding anyone to your form, and in-survey logic adds a second layer of hardening. What remains: the Typeform embed URL is in the respondent's browser after load, and client-side logic can be bypassed by stripping the URL param. The real lock is the wrapper plus uniqueness — a repeat-taker cannot earn a second token regardless of which URL they hit. For a true server-side gate before a single question appears, see [Qualtrics](/survey/qualtrics).
</Info>

## How the layers fit together

The respondent never touches the raw Typeform URL. You distribute the VerifyYou gated link.

1. You set your **gated link** (`app.verifyyou.com/verification/<id>`) as the study URL in Prolific, MTurk, or wherever you recruit.
2. The respondent does the quick liveness and uniqueness check on VerifyYou.
3. Verified? VerifyYou forwards them to your Typeform URL, appending `?vyt=TOKEN` and any pass-through params. Not verified? They return with `?vyc=0` and your wrapper (or return page) routes them to your screen-out URL — they never reach the survey.
4. Inside Typeform, a logic jump checks that `vyt` is present and routes anyone without it to a disqualification end screen. This is browser-side hardening, not a server-side check — it closes the accidental-miss case, not a determined bypass.
5. On submission, Typeform fires a webhook to your endpoint. You validate the token and delete the response via the Typeform Responses API if it is missing or invalid.

The token is a claim stub we mint and look up. It only matches if we made it, it is bound to your study's Flow, and it has not expired (roughly 30 minutes). You cannot guess one. And because uniqueness is baked in, the same person cannot earn a second.

## Step-by-step setup

<Steps>
  <Step title="Create your Verification and get your keys">
    In the VerifyYou dashboard, go to **Verifications → Add verification** to create a new Flow.

    Set the **Redirect URL** — this is where verified respondents land. For Typeform, point it at your wrapper page (see the next step), not directly at the Typeform URL.

    There is no failure-redirect field yet — it is on the build list. Today, a respondent who is not verified comes back to your return page with `?vyc=0` in the API flow, or lands on a VerifyYou end screen on the no-code link. Your wrapper (or return page) is what routes that screen-out to wherever you want it to go.

    From the verification's page, copy:
    - Your **gated link**: `app.verifyyou.com/verification/<id>` — this is what you distribute to respondents.
    - Your **validation key** (`pk_val_...`): a publishable, validate-only key scoped to this one study. It can validate and burn tokens for this Flow; it cannot read participant data or create new verifications. Safe to store in your webhook's environment variables. Do not confuse it with your secret key (`sk_...`), which stays off the client entirely.

    Paste your gated link as the study URL in Prolific, MTurk, Cint, or wherever you recruit.
  </Step>

  <Step title="Pass the token into Typeform">
    VerifyYou returns the token on your redirect URL as `?vyt=TOKEN`. This is where Typeform gets tricky: Typeform hidden fields on a **direct link** are read from the URL's hash fragment (`form.typeform.com/to/FORM_ID#vyt=TOKEN`), not a query string — query params work informally but Typeform does not document or guarantee them. And if you embed Typeform with the Embed SDK, the SDK **strips hash fragments entirely**; an embedded widget only receives hidden fields passed explicitly via `createWidget("FORM_ID", { hidden: { vyt: token } })` or `transitiveSearchParams`.

    The consequence: redirecting straight from VerifyYou into a Typeform URL with `?vyt=TOKEN` does not reliably populate the hidden field, on either a direct link or an embed. The wrapper page is not optional hardening here — it is the required hop. Your redirect URL should point at your own wrapper page, which:

    1. Reads `?vyt=TOKEN` off its own URL (a plain query param on your domain, no Typeform quirks involved).
    2. Peeks the token server-side to confirm it is valid before showing anything.
    3. Either renders Typeform via `createWidget("FORM_ID", { hidden: { vyt: token } })` (embed) or forwards the browser to `form.typeform.com/to/FORM_ID#vyt=TOKEN` (direct link, hash fragment).

    Either way, add a **Hidden field** named `vyt` in Typeform. With the token correctly passed via the SDK's `hidden` option or the hash fragment, it becomes available to logic jumps and to your webhook payload.

    Then add a **Logic Jump** at the very first question: if `vyt` **is empty** → jump to a disqualification end screen. This catches accidental drop-ins without a token. It does not block a respondent who has the Typeform URL and adds a plausible-looking parameter manually — that is what the webhook handles.

    <Warning>
      Do not distribute the raw Typeform share URL in your recruitment message. Anyone with that URL bypasses VerifyYou entirely. Distribute only the VerifyYou gated link — the Typeform URL should appear nowhere in your study instructions.
    </Warning>
  </Step>

  <Step title="Set up the webhook and auto-delete">
    In Typeform, go to **Connect → Webhooks** and point it at your server endpoint.

    When Typeform fires, extract `vyt` from the hidden fields in the payload. Validate the token:

    ```http
    GET /v3/survey/peek?vyt=TOKEN
    Authorization: Bearer pk_val_YOUR_VALIDATION_KEY
    ```

    The peek is idempotent. It checks the token without burning it, so re-checking is always safe:

    ```json
    { "verified": true, "reason": "ok" }
    ```

    Token missing, invalid, already used, or expired (`verified: false`)? Delete the response immediately using your Typeform Personal Access Token:

    ```http
    DELETE https://api.typeform.com/forms/FORM_ID/responses
    Authorization: Bearer YOUR_TYPEFORM_PAT
    Content-Type: application/json

    { "included_response_ids": ["RESPONSE_ID"] }
    ```

    Note that this delete is asynchronous: a `200` means Typeform has registered the deletion request, not that the response is already gone from your results. Do not treat the response as immediately deleted.

    Token valid? Accept the response, then burn the token once so it cannot be reused:

    ```http
    POST /v3/survey/consume?vyt=TOKEN
    Authorization: Bearer pk_val_YOUR_VALIDATION_KEY
    ```

    This is what separates a real backstop from a notification. The bad response never lands in your results CSV.

    <Tip>
      Peek as many times as you need — it never burns the token. Call `consume` exactly once, at the moment you accept a real complete. A respondent who re-submits with the same token is caught on the next peek (now `verified: false`, already consumed) and deleted.
    </Tip>
  </Step>

  <Step title="Test end-to-end">
    Use Test (sandbox) mode in the VerifyYou dashboard. Walk the full flow: open your gated link, complete the liveness check, confirm you land on your wrapper page with `?vyt=TOKEN`, then confirm the wrapper correctly hands the token to Typeform (hidden field populated, either via the embed SDK or the `#vyt=TOKEN` hash fragment on a direct link), submit, and confirm the response appears in Typeform and the webhook fires without deleting it.

    Then try loading the raw Typeform URL with no `vyt` parameter and confirm the logic jump routes you to the disqualification screen. Finally, submit a response from a direct URL with a fake or already-burned token and confirm the webhook deletes it.

    Use Test (sandbox) mode to re-run the whole check as many times as you need while you're building — switch to Live mode only once the flow is solid.
  </Step>
</Steps>

## Why the welcome-screen logic jump is not a gate

Typeform's built-in logic lets you show a welcome screen, then jump to a disqualification end screen if a condition is not met. Teams reach for this as a gate. It is not one.

The welcome screen renders to every respondent before any logic fires. There is no way to block page load on a URL parameter — the respondent sees the first screen, Typeform evaluates the jump condition, and only then redirects. Nothing inside Typeform can perform a server-side token check. A respondent who strips `?vyt=TOKEN` from the URL and loads the raw form directly sees the first question before the jump even has something to evaluate.

The logic jump is useful as a UX signal that the respondent is missing a credential. It is not enforcement. The VerifyYou wrapper is the enforcement layer — it validates before forwarding. The webhook-plus-delete is the cleanup layer — it catches anything that lands anyway.

## Trade-offs at a glance

<CardGroup cols={2}>
  <Card title="VerifyYou wrapper — the primary gate" icon="shield-check">
    Validates server-side before anyone reaches Typeform. Verified → your survey with a token. Not verified → your screen-out URL. This is where most bad respondents are stopped, before a question appears. Works because enforcement is on our side, not inside Typeform.
  </Card>
  <Card title="In-survey logic jump — soft hardening" icon="filter">
    Catches accidental drop-ins who have the survey URL but no token. Browser-side, bypassable by anyone who adds a parameter to the URL. Use it as a second layer, not as the primary gate. Combined with the wrapper, it covers the common case cleanly.
  </Card>
  <Card title="Webhook + auto-delete — the backstop" icon="trash">
    Deletes any submission that arrives without a valid, unused token. Catches the edge cases the logic jump misses. The honest limit: the respondent answered your questions before rejection. For sensitive surveys where even seeing the questions is a risk, consider a wrapper that renders only a confirmation page and never an embedded Typeform.
  </Card>
  <Card title="Uniqueness — the repeat-taker lock" icon="fingerprint">
    Even if a respondent reaches the survey URL directly, they cannot earn a second token. The repeat-taker problem is stopped at the VerifyYou layer regardless of how the survey is configured. One verified human, one response.
  </Card>
</CardGroup>

## Questions teams ask

<AccordionGroup>
  <Accordion title="What happens if a respondent closes the browser mid-survey?">
    The token is burned when your webhook accepts the complete and calls `POST /v3/survey/consume`. If Typeform never fires the webhook (the respondent abandoned before submitting), the token stays valid until its TTL expires — roughly 30 minutes. Within that window, they can re-enter through the gated link and verify again. After expiry, a fresh verification is required.
  </Accordion>
  <Accordion title="Can someone bypass the wrapper by going straight to the Typeform URL?">
    Yes — which is why you should never distribute the raw Typeform URL. The logic jump catches those without a token, but it is browser-side. The webhook catches any submission that comes through without a valid token. And uniqueness means anyone who did get a real token through the wrapper cannot use it twice. These three layers together handle the realistic bypass paths.
  </Accordion>
  <Accordion title="Can I use the same token across multiple surveys in one study?">
    No. A token is bound to one Flow and is single-use. Each survey in a multi-survey study needs its own Flow, and each respondent needs a fresh token per survey.
  </Accordion>
  <Accordion title="What if the webhook fires after the token has expired?">
    The peek returns `verified: false`. Treat it identically to any other invalid token: delete the response. Short surveys rarely hit the ~30-minute TTL, but the webhook handler should cover the case regardless.
  </Accordion>
  <Accordion title="Is the validation key safe to store in my webhook server?">
    Yes. `pk_val_...` is validate-only: it can peek and consume tokens for this one Flow but cannot retrieve PII or create new verifications. Store it as a server-side environment variable for your webhook handler. Do not expose it in client-side JavaScript or embed it in the Typeform form itself.
  </Accordion>
  <Accordion title="Is Typeform the right platform for a study where even seeing the questions is a risk?">
    For that level of sensitivity, Typeform has a real limit: the respondent is inside Typeform before any webhook fires. Consider Qualtrics (which validates server-side before any question block via a Web Service element) or Formspark (where you own the page and can validate before rendering the form at all). See the [platform comparison](/choosing-your-flow).
  </Accordion>
</AccordionGroup>
