## Part 2 — In Alchemer, the soft gate (about 10 minutes, no code)

Level 1 from the guide: capture the token, disqualify anyone without one. Every screen below is Alchemer's real UI.

<Steps>
  <Step title="Capture the token and participant ID as Hidden Values">
    On page 1, add **Action → Hidden Value**, and populate it with the merge code `[url("vyt")]`. Add a second for `[url("pid")]`. Keep page 1 action-only.

    <Frame caption="Real Alchemer UI — the Hidden Value action. 'Populate with the following' takes a merge code; ours is [url(&quot;vyt&quot;)].">
      <img src="/images/alchemer/alchemer-hidden-value.png" alt="Alchemer Hidden Value action editor with the populate field and Select a Merge Code link" />
    </Frame>
  </Step>
  <Step title="Point logic at the URL variable">
    Alchemer's logic builder lists **URL Variable** as a first-class condition, under First Party Data. The rule: if `vyt` is empty, disqualify.

    <Frame caption="Real Alchemer UI — the logic condition picker. URL Variable is a native condition source.">
      <img src="/images/alchemer/alchemer-urlvar-logic.png" alt="Alchemer logic rule condition dropdown showing URL Variable under First Party Data" />
    </Frame>
  </Step>
  <Step title="Send disqualified respondents back to the panel">
    The **URL Redirect** action is built for exactly this: "can be used upon completion, or in combination with disqualification." Set the destination to the panel screen-out URL and flip **Disqualify Respondent** to Yes.

    <Frame caption="Real Alchemer UI — the Actions catalog. URL Redirect: send respondents to an external website, in combination with disqualification.">
      <img src="/images/alchemer/alchemer-redirect-1.png" alt="Alchemer actions catalog highlighting the URL Redirect action" />
    </Frame>

    <Frame caption="Real Alchemer UI — the URL Redirect editor: destination URL, a Disqualify Respondent toggle, and a message for anyone not redirected.">
      <img src="/images/alchemer/alchemer-redirect-2.png" alt="Alchemer URL Redirect action editor with Disqualify Respondent option, URL field, and disqualified message" />
    </Frame>
  </Step>
</Steps>

## Part 3 — The true gate: one-time passwords (Level 2)

The strongest no-code gate on any survey platform. Alchemer's own server refuses entry without a valid, unused password — before a single question renders.

<Steps>
  <Step title="Add the Login/Password action to page 1">
    Page 1, **Add New: Action → Login/Password**. Alchemer recommends it sit alone on the page — it gates everything after it.

    <Frame caption="Real Alchemer UI — adding an Action to an empty page 1.">
      <img src="/images/alchemer/alchemer-add-login-action.png" alt="Alchemer page 1 with the Add New Action link highlighted" />
    </Frame>
  </Step>
  <Step title="Choose one-time passwords">
    Three access modes. The middle one is ours: **"Individual (one-time) passwords that deactivate after use."** Paste the password list from VerifyYou straight into the box, one per line — no CSV needed in this mode.

    <Frame caption="Real Alchemer UI — the three access modes. Note the one-time mode's own wording: passwords 'deactivate after use.'">
      <img src="/images/alchemer/alchemer-login-editor-1.png" alt="Alchemer Login/Password action editor showing the three access type radio options" />
    </Frame>

    <Frame caption="Real Alchemer UI — one-time passwords pasted one per line, or generated by Alchemer.">
      <img src="/images/alchemer/alchemer-login-editor-2.png" alt="Alchemer one-time passwords mode with the Add New Passwords box and Generate Random Passwords link" />
    </Frame>

    <Frame caption="Real Alchemer UI — the spreadsheet mode's settings, including 'Single Use Password — can only be used once' and the note that the action must sit alone on the first page.">
      <img src="/images/alchemer/alchemer-login-editor-3.png" alt="Alchemer Lookup ID/Password mode showing the Single Use Password option and CSV upload" />
    </Frame>
  </Step>
  <Step title="Pre-fill the password from the URL">
    The action's Display tab has **Pre-populate Password**, which takes a merge code. Set it to `[url("vypw")]` and the password the gate handed out arrives already filled in — the respondent types nothing.

    <Frame caption="Real Alchemer UI — Pre-populate Password with a merge code. The example uses a contact merge code; the researcher flow uses [url(&quot;vypw&quot;)].">
      <img src="/images/alchemer/alchemer-prepopulate.png" alt="Alchemer Login/Password Display tab with Pre-populate Username and Pre-populate Password merge code fields" />
    </Frame>
  </Step>
  <Step title="What a blocked respondent sees">
    Wrong password, used password, or no password: an error message the researcher writes, and zero survey content.

    <Frame caption="Real Alchemer UI — the login-failure message. This is the whole experience for anyone without a valid unused password.">
      <img src="/images/alchemer/alchemer-login-error.png" alt="Alchemer error message configuration for failed login" />
    </Frame>
  </Step>
  <Step title="Where the passwords come from (ours to build)">
    VerifyYou mints the pool, hands one password to each verified human, and tracks the states. This is the Invites machinery pointed at a survey.

    <Frame caption="Target-state mockup (COMING) — the one-time password pool: generated, handed to a verified human, redeemed in Alchemer.">
      <img src="/images/alchemer/vy-password-pool.png" alt="Mockup of the VerifyYou one-time password pool screen with statuses and CSV download" />
    </Frame>
  </Step>
</Steps>

## The two things only a live test can settle

<AccordionGroup>
  <Accordion title="When does a one-time password burn?">
    Alchemer's docs don't say whether a single-use password is consumed at login-click or only at survey completion. The Save-and-Continue feature implies burn-at-login. Thirty minutes in a trial account settles it: issue one password, log in, abandon, retry.
  </Accordion>
  <Accordion title="Is the pre-filled login truly zero-click?">
    Pre-populate plus an auto-submitting page-1 should make the gate invisible to verified respondents. The pieces are all documented; the combination isn't vendor-narrated end to end. Worst case, the respondent sees a pre-filled field and clicks one button.
  </Accordion>
</AccordionGroup>
