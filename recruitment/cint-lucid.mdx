Panel traffic from Cint and Lucid often arrives through in-app webviews on mobile, and webviews can be an unreliable place to depend on a browser redirect alone — a disposition URL may fire cleanly in a desktop browser but silently fail to fire on some in-app webviews. That's a general mobile engineering reality, not something unique to any one panel. It's why Cint recommends S2S or callback postbacks for panel and community entry links: redirects alone can fail or be manipulated. The reliable path is a **Server-to-Server (S2S) postback**: your server calls Cint's or Lucid's update endpoint directly the moment you know the outcome, alongside the redirect, as the authoritative disposition report.

HumanCheck slots in between the panel entry link and your survey. It runs liveness first — is a real, live person present right now? — then uniqueness: have we seen this face before? Respondents who pass both carry a token forward. Respondents who fail get a screenout signal sent from your server, not from their phone.

You receive only a **verdict** (pass or fail) and a stable id. No face, no image, no biometric data crosses to you. You are the trusted broker to your client: every complete in your dataset is a real, unique, live human, confirmed before a single question loaded.

<Note>
Terminates are free on both Cint and Lucid. You are charged per complete, not per entry. Screening out a bot, a duplicate, or an unverified respondent costs nothing and does not count against your quota.
</Note>

## How the pieces connect

Two independent legs join at the HumanCheck in the middle. The panel owns recruitment; your survey tool owns the questionnaire. Neither touches the other directly. Your server is the junction.

The panel generates an entry link with a `RID` placeholder — Cint uses `[%RID%]`, Lucid uses `{{RID}}`. You point that link at your server, carry the RID through into a VerifyYou session, and from there the RID rides the whole session and comes back to you on the return URL.

## Steps

<Steps>
  <Step title="Initialize the session">
    Call `POST /v3/initialize` from your server when a respondent arrives, or pre-initialize a batch and embed the resulting URLs directly in the panel entry link.

    Pass the RID in `pass_params` so it returns alongside the token on the redirect URL. Note that `external_tracker` is echoed on webhooks only, not on the return URL — the RID must travel through `pass_params`.

    ```json
    POST /v3/initialize
    {
      "verification_id": "vfy_YOURFLOWID",
      "external_tracker": "cint-study-42",
      "pass_params": { "RID": "[%RID%]" },
      "origin": "https://yourserver.com",
      "return_path": "/verify-return",
      "start_path": "/v/SESSION_ENTRYPOINT"
    }
    ```

    The response contains a `url` pointing to `app.verifyyou.com/v/SESSION`. That is the link the respondent clicks or is redirected to from the panel entry point.
  </Step>

  <Step title="Respondent completes the liveness check">
    The respondent does the quick liveness selfie at `app.verifyyou.com/v/SESSION`. We run liveness first, then uniqueness. On finish, they return to `origin + return_path` carrying:

    - `?vyt=TOKEN` — the single-use verification token
    - `vyc=1` (verified) or `vyc=0` (not verified) — a UI hint only; confirm server-side
    - Any fields from `pass_params`, including `RID`

    Your return endpoint reads all three.
  </Step>

  <Step title="Check the outcome and branch">
    At your return endpoint, read `vyc` but do not trust it alone. Peek the token server-side to confirm it is real. Peek is idempotent and does not burn the token:

    ```http
    GET /v3/survey/peek?vyt=TOKEN
    Authorization: Bearer pk_val_FLOW_RAND
    ```

    Response: `{ "verified": true, "reason": "ok" }`.

    - **`verified: true`**: forward the respondent to your survey with `?vyt=TOKEN&RID=...` in the URL.
    - **`verified: false`** (or `vyc=0`): do not forward. Trigger the S2S postback from your server (next step).

    If `verified` is false even when `vyc` was 1, treat the respondent as a screenout. The token may have expired (~30 min TTL), been tampered with, or already been consumed.
  </Step>

  <Step title="Report screenouts via S2S postback">
    For every respondent who did not pass, call the panel's S2S update endpoint from your server. This runs alongside the browser redirect, as the authoritative disposition report — Cint's docs describe the redirect continuing for session continuity while S2S carries the outcome that actually counts.

    Cint S2S endpoint — the RID goes in the JSON request body, not the URL path:

    ```
    POST https://s2s.cint.com/fulfillment/respondents/transition
    Content-Type: application/json

    {
      "rid": "{RID}",
      "disposition_code": 2
    }
    ```

    Disposition codes: `5` = complete, `2` = screenout, `3` = quota full, `4` = quality termination.

    Lucid S2S endpoint:

    ```
    GET https://[lucid-s2s-endpoint]?supplier_id=XXXX&rid={RID}&status=terminate
    ```

    Both calls come from your server, using the `RID` you carried through from the return URL. Terminates are free — they do not draw quota.
  </Step>

  <Step title="Enforce the token in your survey">
    Your survey's entry point checks the token before questions load (true gate) or on submission (soft gate with webhook delete). See [Choosing your enforcement tier](/choosing-your-flow) and the specific page for your survey tool.

    When a real completion is accepted, burn the token so it cannot be reused:

    ```http
    POST /v3/survey/consume?vyt=TOKEN
    Authorization: Bearer pk_val_FLOW_RAND
    ```

    A consumed token cannot be replayed. Each unique face earns exactly one token per flow, so there is no second attempt.
  </Step>
</Steps>

## Why S2S runs alongside the redirect

Panel respondents on mobile often arrive inside a social-app webview — Facebook, Instagram, TikTok. Those webviews can, as a general matter of mobile engineering, sometimes:

- Swallow a 3xx redirect silently
- Close before the redirect fires
- Block cross-origin navigation in certain states

That's why Cint recommends S2S or callback postbacks for panel and community entry links: redirects alone can fail or be manipulated. S2S does not replace the browser redirect — the redirect still runs for session continuity — but the server-to-server postback is the authoritative disposition report, independent of what the respondent's device does next. The same logic applies to peeking and consuming tokens: confirm outcomes on your server, not from client-side behavior.

## The RID all the way through

The panel appends its RID to your entry link automatically when you use their placeholder syntax. Carry it into `pass_params` at initialization and VerifyYou echoes it back on the return URL. From there your server reads it off the query string and puts it into both the survey forward URL and any S2S postback call.

```
Panel entry link:
https://yourserver.com/start?RID=[%RID%]

Your /start endpoint initializes with pass_params: { "RID": RID_from_query }

Return URL your server receives:
https://yourserver.com/verify-return?vyt=TOKEN&vyc=1&RID=abc123

Survey forward URL:
https://yoursurvey.com/s/STUDYID?vyt=TOKEN&RID=abc123
```

## Trade-offs

<CardGroup cols={2}>
  <Card title="True gate via wrapper page" icon="shield-check">
    Host a thin server page that calls `peek` before rendering the survey link. A respondent with a bad or missing token never sees the survey URL. No bypass path exists. Strongest enforcement.
  </Card>
  <Card title="Soft gate via webhook" icon="webhook">
    Forward the respondent to the survey immediately; your survey platform's webhook calls `peek` on submit and deletes bad responses via the platform API. Respondents enter but bad responses never reach your dataset or count as completes.
  </Card>
  <Card title="S2S postback for screenouts" icon="server">
    Report terminates from your server as the authoritative signal, alongside the browser redirect. Reliable even when webviews cause a redirect to fail. Terminates are free — only completes draw quota.
  </Card>
  <Card title="Honest catch: mid-flow drops" icon="triangle-exclamation">
    If a respondent closes the webview during the liveness check itself, no return signal reaches your server and you have no outcome. This is not unique to HumanCheck — any mid-flow drop is invisible to the panel until a S2S signal arrives. Consider session timeouts if Cint or Lucid requires a disposition on every entry.
  </Card>
</CardGroup>

## About the validation key

The `Authorization: Bearer pk_val_FLOW_RAND` key is a **publishable validation key**. It can peek and consume tokens for one Flow only. It carries no PII and cannot create sessions or access other data. Store it in your server-side credential store.

Do not skip authentication entirely. An open `consume` endpoint lets any caller burn tokens (denial-of-service) and probe for valid token strings (enumeration oracle). The publishable key is scoped narrowly enough to be safe in a server environment; it is not a secret API key.
